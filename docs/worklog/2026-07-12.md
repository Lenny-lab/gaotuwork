## 关键决策与原因

### 决策 1:把 `高途/` 子目录作为独立仓库 `gaotuwork`

- **原状**:git 根目录是 `飞书比赛/`,包含 `topics/`、`archive/`、`TED笔记/` 等无关内容
- **决策**:在 `高途/` 下独立 `git init`,与父目录 git 完全脱钩
- **代价**:丢失了父仓库之前那个 commit `5ffb700 Add standalone AI demo gallery`(它在父仓,跟高途不直接相关)
- **结果**:✅ 父目录的 700+ 个无关文件变动不再影响我们

### 决策 2:用 Flask 统一后端(classmind + feishuapi 合并)

- **原状**:`classmind/api.py` 是 `BaseHTTPRequestHandler`,`feishuapi/python/server.py` 是 Flask
- **决策**:写新的 `app.py`,把两个能力挂在一棵路由树下
  - `/api/*` → classmind 排课核心
  - `/feishu/jssdk_config` → 飞书 JSSDK 鉴权
  - `/feishu/event` → 飞书机器人事件回调
  - `/bitable/webhook` → 多维表格自动化
  - `/healthz` → Render 探针
- **原因**:飞书侧只用配一个公网 URL,部署运维更简单

### 决策 3:选 Render 而不是 Vercel

- **原因**:飞书需要常驻服务 + 事件回调,Vercel 的 Serverless Functions 冷启动 30-50 秒,且不适合长连接
- **Render 免费层**:够用,15 分钟无访问休眠,冷启动 30-50 秒
- **部署时间**:5-10 分钟(OR-Tools 装依赖慢)

### 决策 4:用 iframe 嵌入多维表格(方案 A)

- **三个方案对比**:
  - A. iframe 嵌入(5 分钟,够用)✅ 选了
  - B. iframe + 后端写数据(半小时)
  - C. 三件套全做(半天)
- **选 A 的原因**:用户场景是"在飞书里展示排课数据",iframe 直接显示飞书原生表格,够用且用户能在表格里自由筛选/编辑

## 反思 / 教训

1. **多维表格的 base 链接获取**:飞书 AI 助手默认会把新建表嵌到一个 wiki 文档里,分享链接变成 wiki 形式而不是 `/base/` 直链。后续让用户手动建表,或者提示 AI 把表"移动到云文档根目录",能直接拿到 `/base/` 链接。
2. **H5 可信域名格式**:飞书要求填带 `https://` 的完整 URL,不是裸域名。首次填错会有"请填写有效的 HTTP/HTTPS URL"红字提示。
3. **新版飞书后台的"事件订阅"叫"事件与回调"**:导航菜单找不到"事件订阅"是正常的,要看"事件与回调"那个菜单。
4. **Render 免费层冷启动**:第一次访问 30-50 秒,飞书消息有超时重试机制,生产环境最好升级或用 UptimeRobot 保活。
